다중 센서 기반 에지 IoT 아키텍처: 통신 파이프라인 설계 및 타당성 검토

발표 목적

  • 단순 연결 매체(Medium)의 선정을 넘어, 데이터 특성 및 네트워크 환경에 따른 최적의 프로토콜 탐색
  • [Wear OS] ➔ [스마트폰] ➔ [Jetson Nano] ➔ [서버] 로 이어지는 3구간의 파이프라인 구축 방법론 비교
  • 선정된 방법론의 엔지니어링적 근거 제시 및 아키텍처 효율성에 대한 기술적 논의 진행

통신 구간 정의

  • Phase 1: Wear OS ➔ 스마트폰 (근거리 데이터 동기화 및 스트리밍)
  • Phase 2: 스마트폰 ➔ Jetson Nano (로컬망 기반 대용량 Raw Data 전송)
  • Phase 3: Jetson Nano ➔ 서버 (글로벌 망 기반 추론 결과 비동기 송출)

[Phase 1] Wear OS ➔ 스마트폰 통신 탐색

방법론 탐색: 1) 순수 BLE (GATT 기반 Custom Profile 구축) vs 2) Wearable Data Layer API
추천 및 사유: Wearable Data Layer API (Channel / Message Client 혼용)
  • 고빈도 센서(가속도, 자이로) 데이터를 BLE로 직접 제어 시 발생하는 MTU 제한 및 패킷 Fragmentation 관리 오버헤드를 회피.
  • 운영체제(OS) 단에서 제공하는 RPC 메커니즘을 통해, 백그라운드 상태에서의 데이터 유실 방지 및 자동 재동기화 보장.
💡 기술적 논의점 (Discussion)
  • ChannelClient를 통한 양방향 소켓 스트리밍 시, 스마트폰 측의 데이터 수신 버퍼 최적화 방안은 무엇인가?
  • 연결 단절 시 발생하는 DataItem 동기화 대기열이 스마트폰의 메모리에 미치는 영향과 적정 TTL(Time-To-Live) 설정 기준.

[Phase 2] 스마트폰 ➔ Jetson Nano 통신 탐색

방법론 탐색: 로컬 Wi-Fi 환경 하의 1) TCP Socket vs 2) UDP Socket 전송 계층 비교
추천 및 사유: 트래픽 특성에 따른 혼합 채택 또는 UDP 우선
  • Nano에서의 에지 AI 추론이 최신 데이터에 대한 '실시간 반응성'을 요구하는 경우, TCP의 Head-of-line blocking이 심각한 지연을 초래함.
  • 이에 따라 실시간 센서 스트리밍은 핸드셰이크 오버헤드가 없는 UDP 소켓을 추천하며, 설정 값 등 무결성이 보장되어야 하는 제어 명령만 TCP를 사용하는 이원화 구조 제안.
💡 기술적 논의점 (Discussion)
  • UDP 채택 시 발생하는 패킷 순서 뒤바뀜(Out-of-order) 및 유실을 애플리케이션 계층에서 어떻게 보정할 것인가? (예: 시퀀스 넘버링)
  • 로컬 Wi-Fi 환경에서 TCP의 Congestion Control이 대용량 센서 스트리밍 대역폭에 미치는 실제적 간섭 수준.

[Phase 3] Jetson Nano ➔ 서버 통신 탐색

방법론 탐색: 1) HTTP/REST API (Polling 기반) vs 2) MQTT 프로토콜 (Pub/Sub) + 직렬화 포맷 탐색(JSON vs Protobuf)
추천 및 사유: MQTT + Protocol Buffers(Protobuf) 직렬화 적용
  • 다수의 에지 기기가 서버에 접속할 때 HTTP 헤더의 불필요한 오버헤드를 제거하고, 비동기 Pub/Sub 구조를 통해 서버의 연결 유지 부하(Connection Overhead) 최소화.
  • 외부망(LTE/5G 등)의 불안정성에 대비하여, 애플리케이션 레벨의 구현 없이 MQTT의 QoS 기능을 통해 메시지 도달을 보장.
  • 텍스트 기반 JSON 대신 Protobuf를 사용하여 Payload 크기를 압축, 네트워크 대역폭 극대화.
💡 기술적 논의점 (Discussion)
  • Nano에서 추론한 결과를 전송할 때 적용할 최적의 QoS 레벨(0, 1, 2)은 무엇인가? (QoS 2 적용 시 발생하는 패킷 교환 오버헤드 타당성 검토)
  • 에지 컴퓨팅의 도입으로 줄어든 서버의 트래픽 처리 부하와, MQTT Broker가 감당해야 하는 세션 관리 부하 간의 Trade-off 계산.

종합 토론: 아키텍처 Trade-off 및 확장성 검토

논의 안건 1: End-to-End Latency 관리
  • Wear OS에서 데이터가 생성되어 Nano를 거쳐 서버로 도달하기까지 발생하는 총 지연 시간(Latency)은 각 노드의 어떤 프로세스에서 가장 큰 병목을 일으킬 것으로 예상되는가?
논의 안건 2: 예외 상황 (Network Partition) 대응 정책
  • Phase 2(로컬망) 또는 Phase 3(외부망) 단절 시, 스마트폰과 Nano 중 어느 노드에서 캐싱(Caching)을 주도해야 전체 시스템의 장애 복구력(Resilience)이 높은가?
논의 안건 3: 에지 리소스 최적화
  • Nano는 통신 게이트웨이 역할과 AI 추론 역할을 동시에 수행함. 트래픽 수신(I/O Bound)과 데이터 추론(CPU/GPU Bound) 프로세스 간의 리소스 격리를 위한 소프트웨어 아키텍처(예: 멀티스레딩/비동기 I/O)는 어떻게 설계되어야 하는가?
